2 DllMain入口函数（可选）

2.1 基础实现

2.1.1 参数详解

  • hModule：模块句柄，其实就是DLL在内存的基地址，可以使用该句柄调用相关API获取DLL的路径、名称、大小、资源信息、基地址等信息；
  • ul_reason_for_call：DllMain被调用原因；
值 触发时机 意义 注意事项
DLL_PROCESS_ATTACH 进程首次加载DLL时会收到该通知（仅此一次），不管是隐式加载还是显示加载 初始化 DLL 的全局资源、数据结构，以及为该进程内所有线程做好准备
DLL_PROCESS_DETACH 1-加载进程正常终止卸载该DLL时（最后加载的DLL先收到通知），2-显示调用FreeLibrary使DLL引用计数归零卸载该DLL时 释放所有在 DLL_PROCESS_ATTACH 中分配的全局资源 1-避免使用其他dll（可能在当前dll前面卸载）和自身导出函数， 2-在DLL有多线程执行的情况下，卸载时其他线程可能正在执行DLL代码，会造成访问违规，导出清理接口，清理完所有资源再卸载dll
DLL_THREAD_ATTACH 1-显示调用CreateThread/_beginthreadex创建线程，2-COM组件创建的STA线程，3-线程池的工作线程（首次创建时），4-Windows消息循环创建的UI线程， 5-驱动程序回调线程， CLR管理的托管线程，6-CLR管理的托管线程 初始化与新建线程关联的资源，例如 TLS 数据、线程句柄、线程局部缓存等 1-RtlCreateUserThread创建的线程，由于绕过标准通知机制，不会收到该通知，2-线程池复用线程和纤程(Fiber)转换并非真正线程也不会收到该通知，3-可在收到DLL_PROCESS_ATTACH通知时调用DisableThreadLibraryCalls禁用线程创建和销毁通知提高性能
DLL_THREAD_DETACH 1-线程自然退出（ExitThread/ return from thread func），2-主线程退出且进程未终止，3-使用 FreeLibrary()卸载 DLL，4-使用 FreeLibraryAndExitThread()结束线程，5-使用 ExitProcess()结束整个进程（可能触发） 释放所有在 DLL_THREAD_ATTACH中分配的资源 1-调用TerminateThread()强行 终止线程不会收到该通知
  • lpReserved：保留字段，在不同ul_reason_for_call值下携带的上下文信息。
ul_reason_for_call值 lpReserved值
DLL_PROCESS_ATTACH 非NULL –> 静态连接DLL（即由EXE自动加载），
NULL –> 调用LoadLibrary动态加载
DLL_PROCESS_DETACH 非NULL –> 加载进程正在退出（系统自动卸载DLL），
NULL –> 调用FreeLibrary主动卸载
DLL_THREAD_ATTACH/ DETACH 始终为 NULL，无特殊含义

2.1.2 返回值

  • 当 ul_reason_for_call值为DLL_PROCESS_ATTACH时（返回值有实际意义）
    • 返回TRUE表示DLL初始化成功，加载成功；
    • 返回FALSE表示DLL初始化失败，加载失败，立即调用 DLL_PROCESS_DETACH（此时 lpReserved 参数非NULL）；对于显式加载(LoadLibrary)：LoadLibrary 返回 NULL；对于隐式加载：进程启动失败。
  • 当ul_reason_for_call值为DLL_PROCESS_DETACH/DLL_THREAD_ATTACH/DLL_THREAD_DETACH时返回值没有实际意义，会被系统忽略，但是还是建议返回TRUE。

2.2 DllMain编写时的注意事项

2.2.1 不要在DllMain里加载和卸载其他DLL

禁止函数 原因
LoadLibrary / FreeLibrary Loader正在持锁，递归加载会导致死锁
CoInitialize / CoCreateInstance 可能加载 COM DLL，风险同上
创建 COM 对象 / 启用 WMI / 使用 ShellExecute等 可能隐式加载系统 DLL，如 shell32.dll、oleaut32.dll

2.2.2 不要执行等待/堵塞操作

禁止操作 原因
WaitForSingleObject、Sleep、ReadFile 等 Loader 正在初始化 DLL，阻塞会卡死整个进程

2.2.3 所有初始化必须是“快速、安全、局部”的

只允许做如下操作：

  • TLS索引申请（TlsAlloc）
  • 分配DLL自己的全局资源
  • 写日志（非堵塞）

2.2.3 在DllMain中使用创建线程时需要注意以下问题

禁止操作 原因
不要在DllMain中等待线程完成（如调用WaitForSingleObject等） Loader 会死锁
不要在新线程函数内调用 DLL 导出函数访问未初始化数据 可能访问野指针、NULL、崩溃
不要在新线程函数内加载和卸载其他DLL 会触发递归 DllMain，Loader 锁未释放，死锁

2.2.4 不能调用malloc/free、new/delete、STL容器操作

  • 加载器锁（Loader Lock）的限制：

    • DllMain 是在加载器锁的上下文中执行的，而 new/delete 或 STL 容器的操作可能依赖 堆管理器（Heap Manager）或 CRT（C Runtime Library）初始化。
    • 如果这些操作尝试获取另一个锁（如堆管理器的锁），而该锁已经被其他线程持有，可能会导致 死锁（Deadlock）

  • CRT（C Runtime Library）可能未初始化，此时调用 new/delete或 STL 操作可能导致未定义行为

  • 线程安全问题，DllMain`是在加载器锁的保护下执行的，但某些 STL 实现（如 MSVC 的 STL）可能使用线程局部存储（TLS）或 全局锁，这可能导致：

    • 死锁（如果锁已被其他线程持有；
    • 数据竞争（如果 STL 内部状态未正确同步）。

  • 可能导致递归加载，如果 new/delete 或 STL 操作触发了另一个 DLL 的加载（例如，某些调试版本的 STL 可能加载额外的调试 DLL），则会导致 递归进入 DllMain，而 Windows 的加载器锁不支持递归，可能导致崩溃

注意：STL 容器内部会调用new/delete。

3 开发稳健DLL的建议

3.1 设计阶段

3.1.1 接口尽量简单、明确

  • 尽量只导出C风格（extern “C”）接口，避免C++名字修饰
  • 若需要导出类，建议适应纯虚接口 + 工厂函数模式（详见COM风格设计）

3.1.2 版本兼容性

  • 设计获取DLL版本导出接口
  • 不要随意修改已有导出函数的签名，新增函数而非替换。

3.1.3 封装内存管理

  • 尽量不要将内存分配和释放的责任交给外部调用，由DLL提供配套内存管理接口(可以导出)。

3.2 实现阶段

3.2.1 慎用全局对象/单例

  • 全局对象的构造顺序和析构顺序不可控，容易导致崩溃；
  • 推荐显式初始化+释放机制，或使用懒加载的线程安全静态局部变量。

3.2.2 DllMain 编写注意

详见：2.2 DllMain编写时的注意事项。

3.2.3 线程安全

  • 注意静态数据、多线程读写同步
  • 使用CRITICAL_SECTION、std::mutex、InterlockedXXX系列函数加速。

3.2.4 错误处理机制

  • 提供统一的错误码或异常机制（建议导出 C 接口时不抛异常）；

  • 提供日志记录机制（内部使用，帮助调试）。

3.3 运行和部署阶段

3.3.1 避免跨模块内存管理

  • DLL中分配的内存应由DLL释放，不要在外部调用释放；
  • 特别注意：若DLL和外部调用使用不同CRT（运行库），delete会崩溃。

3.3.2 使用正确的运行库

  • 尽量统一使用/MD或/MT编译选项；
  • 使用/MD选项时，DLL不应和外部调用混用new/delete。

3.3.3 避免资源泄漏

  • DLL卸载时释放所有句柄、内存、线程、钩子等资源。
